# Effort Moat


Honour what people have built, so their progress remains useful and recognisable.

## What it is

Effort Moat makes a user’s time, choices, creations, and personal setup visible and valuable. Saved work, tailored defaults, histories, collections, and progress views help the product feel like *their* place rather than a generic tool.

## Why it works

People value progress they can see and reuse. The IKEA / endowment dynamic, and the line between continuity and hostage data, are defined in [Investment and Continuity](../concepts/13-investment-and-continuity.md).

This card is the *honouring* half: reflect saved work, restore context on return, and make portability easy. Eliciting the contribution itself is [Investment](investment.md)—do not re-ask for busywork that only raises switching costs.

## When to use it

- After users create, organise, configure, or contribute something meaningful
- When preferences, history, or saved work can make the next session better
- In re-entry moments where a useful recap can restore context
- When Personalisation can make the product adapt to explicit user choices

## Do

- Reflect meaningful investment through saved work, timelines, recaps, and useful defaults
- Make completed setup immediately pay off in the experience that follows
- Preserve user input through autosave, version history, and clear recovery paths
- Make portability and account controls easy; Trust Building strengthens attachment more than lock-in does

## Don't

- Add tedious setup merely to create commitment
- Use streak loss, badges, or social status to make leaving feel costly
- Hide, delete, or hold user-created work hostage
- Treat accumulated data as proof of loyalty when it no longer serves the user

## Founder Tip

The strongest moat is not “you cannot leave”; it is “leaving means giving up something genuinely useful you chose to build here”.

## Make It Yours

1. **Map meaningful investment**
   - What do users create, curate, learn, or configure that improves future use?
   - Which effort is currently invisible once the task is complete?

2. **Make the payoff immediate**
   - What useful result appears as soon as setup or contribution is finished?
   - Can a new default, saved view, or recommendation demonstrate that the product learned something helpful?

3. **Protect ownership**
   - Can users retrieve drafts, undo accidental changes, export their work, and understand how it is used?
   - Where could a loss of data or context turn investment into distrust?

4. **Check for real value**
   - Do users with more saved work get better outcomes, or are they simply more locked in?
   - What would make a first-time user want to build a little more because the benefit is clear?
